## Rationale

**T001–T002 amend feature 0030 domain models.** `ScenarioSummary` (defined in 0030 T001) lacks `TestType` and `ScreenshotUris`, and `TestResult` (0030 T002) lacks `ScenarioResults` and `RawCommentMarkdown`. These fields are written by `ReportWriter` and read by this feature's portal API. Since 0030 is not yet implemented (Implement: Pending), extending the entities here is additive and non-breaking — Cosmos is schema-less and the infrastructure repository uses explicit field mapping. The ReportWriter implementer must treat T001–T002 here as prerequisites and populate the new fields. These tasks are placed first so any downstream code in this feature compiles against the complete type surface.

**`TrendPoint` and `RunHistoryItem` as dedicated read-model types (T003–T004).** These carry only the fields the portal views need — they are not full entity objects. Defining them in `Testurio.Core` keeps DTOs in `Testurio.Api` thin (near-1:1 mappings) and keeps the `IStatsRepository` interface typed rather than returning raw `IEnumerable<TestResult>` that the App layer must project.

**`IStatsRepository` extended rather than a new read interface (T005).** `ITestResultRepository` (feature 0030) is the write-path interface for `TestResults`. The read path for portal statistics belongs in `IStatsRepository` (introduced in feature 0010) — the interface designed for portal-facing projection queries. Adding the two new method signatures here avoids creating a competing read repository for the same Cosmos container and keeps read logic centralised in `StatsRepository`.

**Project ownership validation inside the repository (T006–T007).** Both history methods validate that `projectId` belongs to `userId` by querying the `Projects` Cosmos container before touching `TestResults`. This keeps the ownership check inside the infrastructure boundary; the App layer sees a clean nullable return and does not repeat the check.

**Trend computation server-side in T006.** The 90-day bucket reduction happens in the API layer rather than on the client, even though the client loads all runs (Q3: client-side filter for the list). This keeps the response payload for the chart fixed at 90 integers regardless of how many runs exist, while the full `RunHistoryItem` list is still sent for the table. The client uses the pre-bucketed `trendPoints` directly and filters by range without an additional request.

**Frontend follows ui.md canonical layer order (T012–T022).** Types (T012) compile before the service (T013), which compiles before the hooks (T014). The MSW handler (T015) is registered before any component renders. Within components, leaf components are implemented before composites: `TrendChart` (T016) and `RunHistoryTable` (T017) are direct children of the page; `ScenarioCard` (T018) is a leaf used by `RunDetailPanel` (T019); `RunDetailPanel` is composed into `ProjectHistoryPage` (T020). Translation keys (T021) and route registration (T022) are last because they depend on knowing the final component names and string keys.

**`RunDetailPanel` uses `useRunDetail` directly (T019).** The panel fetches its own data rather than receiving it as a prop, enabling `enabled: Boolean(runId)` — no network request fires until a row is clicked. This avoids preloading all run details on page mount and keeps `ProjectHistoryPage` free of per-run fetch logic.

**`@mui/x-charts` for `TrendChart` (T016).** The project already uses MUI; `@mui/x-charts` is the native MUI chart library and requires no additional design-system integration work.

**Cross-feature dependencies.**

- **Feature 0010**: `RunStatusBadge` (reused in T017 for verdict rendering); `IStatsRepository` and `StatsRepository` (extended at T005–T007); route constants `PROJECT_HISTORY_ROUTE` and `PROJECT_SETTINGS_ROUTE` (consumed in T020 and T022, not redefined); `StatsEndpoints.cs` (two new endpoints added at T011)
- **Feature 0029**: `StepExecutionResult.ScreenshotBlobUri` — screenshot URIs flow into `ScenarioSummary.ScreenshotUris` when `ReportWriter` constructs `ScenarioSummary` from `UiE2eScenarioResult`
- **Feature 0030**: `ScenarioSummary` (extended at T001), `TestResult` entity (extended at T002), `ITestResultRepository` (write path, consumed not modified here); `ReportWriter` must populate the new fields; this feature must be implemented after 0030's Domain tasks (T001–T002 of 0030) are complete
- **Feature 0043**: Real-time run status updates on the history page are not in scope for MVP; the history page renders a static snapshot only
