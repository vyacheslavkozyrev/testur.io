## Rationale

**Domain before everything else.** T001 (`ApiAuthMethod` enum) and T002 (`ApiAuthApiKeyPlacement` enum) are the primitive vocabulary that all other types in this feature reference. They must exist before the `Project` entity extension (T003), the credentials union (T004), and the provider interface (T005) can be written. T003 extends `Project` with the seven new fields that are the single source of truth for auth configuration; this extension must be stable before any service, DTO, or executor can read from it. T004 (`ApiTestAuthCredentials`) is the handoff type between the infrastructure layer and `HttpExecutor` — defining it in domain means neither side imports the other. T005 (`IApiTestAuthCredentialProvider`) declares the contract that infrastructure implements and the executor consumes, with no circular dependency.

**Infrastructure before API and pipeline.** T006 extends `ProjectSecretNamespace` — an existing infrastructure file — with the three new secret key constants. Centralising secret naming here prevents naming drift if the constants are referenced by both the API service (writes secrets) and the credential provider (reads secrets). T007 (`ApiTestAuthCredentialProvider`) depends on T005 (the interface), T006 (the secret key constants), and the existing `ISecretResolver` abstraction already registered in 0017. T008 registers the new interface in DI before any consumer (API service, executor) resolves it.

**DTOs and service before endpoints.** T009 (`ProjectApiAuthDto`) and T010 (`UpdateProjectApiAuthRequest`) are defined before the service (T011) so the service can reference them directly without forward declarations. T011 (`ProjectApiAuthService`) implements the business logic for ownership validation, Key Vault secret writes, and mode-switch secret clearing — it must be stable before the endpoint wires it in. T012 registers the two new endpoints (`GET` and `PATCH /v1/projects/{projectId}/api-auth`), delegating entirely to `ProjectApiAuthService` via the existing `ValidationFilter<T>` pattern. No new route group or authentication middleware is required — the existing `/v1` group with `RequireAuthorization()` covers both endpoints.

**`HttpExecutor` extension after domain and infrastructure.** T013 extends `HttpExecutor` to resolve credentials once per run via `IApiTestAuthCredentialProvider` and apply them to every request. This task is placed after T001–T008 because it depends on the `ApiTestAuthCredentials` union (T004) and the provider interface (T005). It is placed before the UI layer because it is a backend pipeline concern independent of the portal experience. `PlaywrightExecutor` is not touched — this feature governs API test execution only, as stated in Out of Scope.

**`ProjectService` update (T014) is a narrow, additive change.** `CreateAsync` must initialise `ApiAuthMethod` to `None` (AC-030) so that existing project creation logic (0006) produces valid project documents. `ProjectDto` and `ToDto` need to expose `apiAuthMethod` so the portal can show the current method on load (US-005). This task is ordered after T003 (entity extension) and before the UI layer so the portal's React Query hooks receive the correct shape.

**Separation from feature 0017 (environment access).** Feature 0017 configures how the worker *reaches* the staging environment (IP allowlist / Basic Auth / header token at the network layer). Feature 0023 configures how `HttpExecutor` *authenticates individual API test requests* against the product's API (Bearer token / API key / Basic Auth at the application layer). These are orthogonal concerns stored in separate fields on the project document and handled by separate providers and endpoints. The API Authentication section in `ProjectSettingsPage` is a distinct section from the Testing Environment section added by 0017.

**Re-use of existing `CredentialRetrievalException`.** Feature 0017 already introduced `CredentialRetrievalException` in `Testurio.Core`. T007 throws the same exception when Key Vault is unreachable — no new exception type is needed, and catch sites in the worker already handle it.

**UI follows canonical order.** T015–T021 follow the order mandated by `ui.md`: types → service → hook → mock → component → page → i18n. A separate `projectApiAuth.json` locale file is used rather than extending `project.json` to keep domain-specific translation keys scoped and avoid merge conflicts with the project settings keys from 0006, 0008, 0017, and 0022.

**No new route registration.** The two new endpoints are added to the existing project endpoint group (`/v1/projects`). No new route file is created.

**Cross-feature dependencies:**

- **Depends on 0006 (Project Creation & Core Configuration):** `Project` entity, `ProjectRepository`, `ProjectService`, and project endpoint group must exist. T003 and T014 extend them minimally with no breaking changes to existing fields.
- **Depends on 0017 (Testing Environment Access Configuration):** The `ISecretResolver`, `ProjectSecretNamespace`, and `CredentialRetrievalException` established in 0017 are reused directly. The existing `ProjectAccessCredentialProvider` pattern is the direct model for T007. `HttpExecutor` (extended in 0017 for environment credentials) is extended again in T013 for request-level auth credentials — both layers coexist independently in the executor.
- **Depends on 0029 (Executor Router):** `HttpExecutor` must exist before T013 can extend it. `ProjectConfig` — the struct passed into `HttpExecutor` — carries the `Project` data that includes the new auth fields.
- **Does not block any other MVP feature directly**, but completes the "Bearer token only for POC" stub explicitly called out in the 0003 out-of-scope section.

**No Cosmos DB migration.** Cosmos DB is schema-less; the new fields on `Project` are additive. Existing project documents that lack them default to `ApiAuthMethod.None` via the `?? ApiAuthMethod.None` fallback in `ProjectApiAuthService` and `ApiTestAuthCredentialProvider` (AC-036).

**Tests last, per QA rules.** Unit tests (T022–T024) mock `ISecretResolver`, `IApiTestAuthCredentialProvider`, and `IHttpClientFactory` to verify all acceptance-criteria branching paths without live external calls. Integration tests (T025) exercise the full validation and persistence pipeline through the API. Frontend component tests (T026) verify field rendering, masking, pre-fill, validation, and hook wiring.
